## 1. プロダクト概要

各国特許庁から届く拒絶理由通知（Office Action, 以下 OA）への応答を、Claude API を用いて支援する社内 Web アプリ。
ユーザー（企業知財部員・弁理士）が、本願文献・OA文書・引用文献・現クレームをアップロードし、**Claude チャットライクな UI** 上でステップを進めることで、以下を半自動で得られる。

1. 各文書の全文テキスト化と内容要約
2. 各拒絶理由の妥当性評価（請求項×構成×引用文献のチャート）
3. 応答方針案（複数）
4. 代表補正案 → 全文補正案
5. 意見書案（エストッペル配慮）
6. 補正書・意見書・見解書の Word 出力

**位置づけ**: 既存の有償 SaaS で提供される同種機能を、社内で内製・無償運用するための代替。

## 2. ゴール / 非ゴール

### ゴール（このプロダクトが達成すること）
- OA 受領後の検討時間を大幅短縮する（理解・妥当性判断・方針検討・書面作成）
- 全プロセスを 1 つのチャット UI 上で、コンテキストを引き継ぎながら進められる
- 案件単位で履歴を保存し、途中から再開できる
- 補正後もできるだけ広い権利範囲を保つ方針を複数提示する
- 各国の補正運用ルールを考慮する

### 非ゴール（やらないこと・MVP に入れないこと）
- 各国特許庁ページ（例: 米 Patent Center）からの OA / 引用文献の自動取得 → **やらない**（要求仕様で明記済み）
- 出願管理・期限管理・年金管理などの IP 管理機能
- 法的最終判断の代替（本ツールはあくまで弁理士の検討支援であり、出力は人間がレビューする前提）
- リアルタイム共同編集

## 3. スコープとフェーズ

「全部を一度に作らない」。最小で価値の出る縦の一本（vertical slice）を先に通し、その後ステップを足す。

### Phase 0: 基盤（最優先）
- リポジトリ / Next.js + Tailwind セットアップ、Vercel デプロイ確認
- Supabase 接続、Auth（Google + Email/Pass）
- DB スキーマ + Storage + RLS の最小構成
- Claude API をサーバーサイド経由で 1 回呼べる疎通確認（「Hello」往復）

### Phase 1: MVP（縦の一本を通す）
- ログイン → 案件新規作成 → ファイルアップロード（Step1）
- Claude による全文テキスト化＋要約（Step2-3）をチャットに表示・DB保存
- チャット入力枠（テキスト + ファイル添付 + 送信）が機能
- 案件リスト表示・再開・削除・**お気に入り登録/解除**（CRUD）
- ここまでで「アップロードして要約が返ってくる」体験が成立すること

### Phase 2: 検討フロー全ステップ
- Step4-5: 妥当性評価（チャート）
- Step6-7: 応答方針案（複数）★最重要
- Step8-9: 代表補正案 ★重要（必要最小限の補正）
- Step10-11: 全文補正案
- Step12-13: 意見書案
- Step14: 補正書 / 意見書 / 見解書の Word 出力
- 各ステップで自由に追問できる（コンテキスト継続）
- **オートラン（自動連続実行）**: Step1 から指定ステップまで確認なしで連続実行（§7.10）

### Phase 3: あったら良い機能（任意・後回し）
- 引用文献が無い場合の Web 検索（Claude の web_search ツール / Google Patents 参照）
- 対象国ルールの Web 検索による補強
- サブエージェントによる多段検討
- レスポンシブ最適化の作り込み、UI 磨き込み
- プロンプトキャッシュによるコスト最適化（§7.5。利用パターン次第で効果が出る場合のみ）

## 4. ユーザー / ユースケース

- **主ユーザー**: 企業知財部員、特許事務所の弁理士。複数国の OA を扱う。**MVP は個人利用前提**（本人のみが自分の案件を参照）。良いアプリになれば将来的に共有機能を検討。
- **重点対応国**: **JP / US / EP / WO / CN**。補正・主張は各国の運用ルールをある程度考慮する（精度は段階的に向上）。
- **代表ユースケース**: 「US の OA が届いた。引用文献2件と本願公報、現クレームを入れて、妥当性を確認し、できるだけ広い補正方針を比較したうえで、補正書・意見書のドラフトまで作りたい」
- **利用環境**: PC（デスクトップ中心）。モバイルは閲覧・軽操作レベルで崩れなければよい（レスポンシブ対応はするが最適化優先度は低）。

## 5. システムアーキテクチャ

```
[Browser (Next.js Client Components / Tailwind UI)]
        │  ① 認証(Supabase JS) / ② ファイルupload(Supabase Storage) / ③ チャット送信(fetch)
        ▼
[Next.js Route Handlers / Server Actions  (Vercel)]   ← ★ここでのみ ANTHROPIC_API_KEY を使用
        │  ④ 抽出テキスト・履歴を取得         │  ⑤ Claude API 呼び出し
        ▼                                    ▼
[Supabase: Postgres(RLS) + Storage + Auth]   [Anthropic Claude API]
```

設計原則:
- **Claude API キーは絶対にクライアントに出さない**。API 呼び出しはすべて Next.js のサーバー側（Route Handler / Server Action）で行う。`ANTHROPIC_API_KEY` は Vercel の環境変数（サーバー専用）に置く。`NEXT_PUBLIC_` 接頭辞は付けない。
- Supabase の anon key はクライアントで使ってよいが、**RLS を必ず有効化**してユーザーは自分のデータのみアクセス可能にする。
- 重い処理（Claude 呼び出し）はサーバー側でストリーミング（後述）し、クライアントは逐次表示する。

## 7. Claude API 連携設計（最重要）

### 7.1 呼び出し方式
- すべてサーバー側。クライアントは `/api/chat`（仮）に「案件ID・ステップ・ユーザープロンプト・添付参照」を送る。
- サーバーは DB から「これまでのコンテキスト（過去メッセージ・抽出テキスト・直近成果物）」を組み立て、システムプロンプト（§11）と共に Claude へ送る。
- 公式 SDK を使用。モデル文字列は環境変数 or 設定ファイルで一元管理し、ステップごとに切替可能にする。

### 7.2 利用モデル（実行時）
> 開発作業の Claude Code とは別。これは「アプリが実行時に叩くモデル」。**API 課金はトークン従量で、Claude.ai の Max プラン契約とは独立**（Max を解約してもアプリ実行コストには影響しない）。

- **既定は全ステップ `claude-sonnet-4-6`**（コスト重視。PO 方針）。
- ただし本アプリの中核である **妥当性評価(S4)・応答方針(S6)・補正案(S8/S10)** は品質が重要。Sonnet で品質が不足する場合は、これらのステップのみ `claude-opus-4-8` に上げられるよう、**「ステップ → モデル」を設定で一元管理し容易に切替可能**にしておく。
- まずは全 Sonnet で構築・評価し、必要に応じて重要ステップを Opus に切替える運用とする。

### 7.3 ファイル処理戦略（重要）
- **PDF**: Claude API は PDF を **ネイティブに**読める（テキスト＋図表を vision で処理）。テキストレイヤの無いスキャン PDF も vision で読み取れるため、**別途 OCR ライブラリは原則不要**。`document` ブロック（base64 または Files API の `file_id`）で渡す。
- **Word(.docx) / .txt / .md / .csv**: これらは document ブロック非対応。**サーバー側でプレーンテキストに変換**してメッセージ本文に含める（.docx は `mammoth` 等）。
- **制限に注意**: PDF は概ね **100ページ / リクエスト全体32MB** の上限がある。特許明細書＋複数引用文献は容易に超える。対策:
  - **Step1 で全 PDF を Files API にアップロード**し `file_id` を保存（§7.4）。
  - **Step2 で各ファイルを個別に全文テキスト化**し、抽出テキストを Supabase に保存する。
  - **Step4 以降は原本 PDF を毎回送らず、保存済みの抽出テキストを送る**（ページ上限回避＋トークン削減）。図面が判断に重要な場面のみ、該当文書を `file_id` で再参照する（§7.5）。
  - 1ファイルが100ページ超の場合は分割して読み込む。

### 7.4 Files API（積極的に利用する）
本アプリの基本方針として **Files API**（beta ヘッダ `files-api-2025-04-14`）を積極利用する。Files API はアップロードしたファイルを `file_id` で参照でき、**明示的に削除するまで保持**される（保存先は API キーのワークスペース単位。時間による自動削除はない）。

運用ルール:
- **Step1 のアップロード時に、全 PDF を Files API にアップロードし、返ってきた `file_id` を Supabase（`case_files.anthropic_file_id`）に保存**する。
- 以降 Claude に**原本（図面含む）を見せたい場面では `file_id` で参照**する（base64 を毎回送らない）。特に図面が要旨となる引用文献の検討で有効。
- 案件の削除時には、対応する `file_id` も DELETE エンドポイントで削除し、ストレージを浪費しない。
- Word(.docx) / .txt / .md / .csv は Files API の document 参照に使わない（§7.3）。サーバー側でテキスト抽出して扱う。

### 7.5 コンテキスト管理 & コスト方針（ステップ横断）
本アプリは「妥当性評価結果 → 応答方針 → 補正 → 意見書」と前段の成果物を引き継ぐため、各ステップのコンテキストが肥大化しやすい。**コスト最適化の主軸は「抽出テキストの Supabase 保存と使い回し」**とする。

方針（優先順位順）:
1. **テキスト抽出 → Supabase 保存 → 使い回し（主軸・MVP で必須）**: Step2-3 で各文書を全文テキスト化し `case_files.extracted_text` に保存。**Step4 以降は原本 PDF を毎回送らず、保存済みの抽出テキストをコンテキストに含める**。これによりページ上限回避＋トークン削減を確実に達成する。図面が判断に重要な場面のみ、該当文書を `file_id`（§7.4）で再参照する。
2. **成果物の構造化保存**: 各成果物（要約・妥当性結果・方針案・補正案・意見書案）を DB に保存し、次ステップでは「必要な成果物＋抽出テキスト＋新規ユーザープロンプト」を再構築して送る。
3. **長文化対策**: 文脈が長くなりすぎる場合は古い往復を要約して圧縮する（Phase2 後半で検討）。
4. **プロンプトキャッシュ（モードで使い分け）**:
   - **手動モード（1ステップずつ熟考）**: ステップ間が5分〜数十分空くため TTL 切れでほぼ無効。**採用しない**。
   - **オートランモード（§7.10）**: Step2→7 を短時間に連続実行し、同じ大きな文書コンテキストを使い回すため**キャッシュが有効**。**この場合のみ採用してよい**。実装する場合は、不変の大きな塊（抽出テキスト等）を `cache_control` のプレフィックスに置き、`ttl: "1h"` を明示する（デフォルト5分は連続実行でも切れやすいため）。
   - 効果が見込めない使い方では無理に使わない（書き込み料だけ払う損を避ける）。Phase2 以降で実測しながら判断。

### 7.6 Citations（推奨・任意）
- document ブロックで Citations を有効化すると、回答が原文のどのページ/箇所に基づくか追跡できる。特許の妥当性評価では**根拠トレーサビリティ**が有用なので、Step4（妥当性評価）で活用を検討する。

### 7.7 Word 出力（Step14）
- **Claude にバイナリ(.docx)を作らせない**。Claude には「補正書・意見書・見解書の構造化テキスト（見出し・本文）」を出力させ、**アプリ側（サーバー）が `docx` ライブラリで .docx を生成**する。
- **社内テンプレは無し**。補正書・意見書は**各国（JP/US/EP/WO/CN）のフォーマット・記載ルールに従う**こと。
- 生成ファイルは Storage に保存し、署名付き URL でダウンロードさせる。
- 見解書には「本願概要・OA概要・引用文献概要・妥当性評価結果・複数応答方針案・最終決定方針」を含める。

### 7.8 機密性 / データ取り扱い
- 本アプリは**個人アカウントでの利用**。OA 対応でアップロードするのは**主に公知（公開済み）情報で機密情報を含まない**前提のため、**ZDR は不要**、クラウド API の利用に問題はない。
- Anthropic API は既定で API 入力をモデル学習に使用しない（標準で短期間のみ保持）。
- 注意（前提が変わる場合）: 未公開・係属中の機密書類を扱うことになった場合は、データ取り扱い（ZDR 等）を再検討する。
- ログに文書本文を不必要に残さない（最小限のエラーログに留める）。

### 7.9 ストリーミング & エラーハンドリング
- Claude 応答は `stream: true` で受け、サーバー → クライアントへ逐次転送（Claude チャットの体験を再現）。
- API 失敗（レート制限・タイムアウト・サイズ超過）はユーザーに分かる形でメッセージ表示し、リトライ可能にする。送信途中のユーザープロンプトは失っても再入力できるようにする。

### 7.10 オートラン（自動連続実行）モード ★重要要件
ユーザーが各ステップで都度確認しなくても、**Step1 のアップロード後に Step2→…→（指定ステップ）まで一気に自動で進められる**ようにする。**各ステップの検討は省略せず確実に実行**し、ユーザー確認だけをスキップする。

- **トリガー**: 入力枠の自由入力（§10 の判定）で「応答方針案まで一気に出して」等と指示されたら、サーバー側オーケストレータが対象ステップまで連続実行する。
- **標準の停止点**: **応答方針案の出力（本 PRD の Step7）**。ここで一旦停止し、ユーザーにレビュー・方針決定を促す。指示により停止点は可変（例: 妥当性評価まで／代表補正案まで）。
- **動作**: Step2(解析)→Step4(妥当性)→Step6(方針) の LLM 呼び出しを順に実行し、各成果物を逐次 DB 保存＆チャットに**ストリーミング表示**（Step3/5/7 の表示に相当）。各ステップは正規のシステムプロンプト（§11）で**フル分析**する。
- **コスト/速度**: 連続実行のためプロンプトキャッシュが有効（§7.5）。同一の大きな抽出テキスト・コンテキストを使い回す。
- **エラー時**: 失敗したステップで停止し、どこで止まったかを表示。ユーザーはそのステップから再開できる。
- **手動モードとの両立**: 1ステップずつ確認しながら進める従来モードも維持する（オートランは上乗せ機能）。

## 8. データモデル（Supabase）

> 全テーブルで RLS を有効化し、`auth.uid()` に基づき本人データのみ参照・変更可能にする。

### 8.1 テーブル

**profiles**（auth.users と 1:1）
- `id` uuid PK（= auth.users.id）
- `email` text
- `full_name` text
- `created_at` timestamptz

**cases**（OA 対応案件）
- `id` uuid PK
- `user_id` uuid FK → profiles.id
- `publication_number` text（本願公報番号。案件識別に使用）
- `country` text（対象国。JP / US / EP / WO / CN 等）
- `title` text（リスト表示用。未定なら公報番号）
- `current_step` int（進捗ステップ 1–14）
- `status` text（'active' / 'archived'）
- `is_favorite` boolean（既定 false。サイドバー三点メニューから切替。お気に入りは上部に表示）
- `created_at` / `updated_at` timestamptz

**case_files**（アップロード文書のメタ＋抽出テキスト）
- `id` uuid PK
- `case_id` uuid FK → cases.id
- `doc_role` text（'applicant'本願 / 'oa' / 'reference'引用 / 'claims'現クレーム）
- `file_name` text
- `file_type` text（pdf/docx/txt/md）
- `storage_path` text（Storage 上のパス）
- `anthropic_file_id` text（Files API の file_id。Step1 で取得・保存。案件削除時に DELETE する）
- `extracted_text` text（全文テキスト。Step2-3 で格納）
- `summary` text（要約）
- `created_at` timestamptz

**messages**（チャット履歴＝ステップ進行の本体）
- `id` uuid PK
- `case_id` uuid FK → cases.id
- `role` text（'user' / 'assistant'）
- `step_no` int（どのステップでのやり取りか）
- `content` text（ユーザープロンプト or LLM 出力）
- `created_at` timestamptz

**case_artifacts**（任意・主要成果物の構造化保存。再開・再利用を容易に）
- `id` uuid PK
- `case_id` uuid FK
- `kind` text（'summary' / 'validity' / 'strategies' / 'rep_amendment' / 'full_amendment' / 'opinion' / 'docx'）
- `step_no` int
- `payload` jsonb（または text）
- `created_at` timestamptz

### 8.2 Storage
- private バケット `case-files`。パス規約 `{user_id}/{case_id}/{filename}`。
- アクセスは署名付き URL。生成 Word も同バケットに保存。

## 9. 画面要件

> デザインは **Claude チャットライク**。視覚設計は **Claude Design** を正とする（運用は §9.0）。

### 9.0 デザイン運用（Claude Design + /design-sync）

## 10. ステップ別詳細フロー（処理仕様）

## 11. システムプロンプト集

> 下記は要求仕様のプロンプトを土台に整理したもの。**ファイルやコードに定数として一元管理**し、実装中に改善できるようにする。実際の文言はチューニング前提。各国の補正運用ルールは Claude の知識＋（Phase3 で）Web 検索で補強する。

- **S2（解析）**: 「あなたは特許 OA 対応の専門家です。添付ファイルを全て読み込み、本願発明・OA文書・各引用文献・(あれば)現クレームについて、各々の全文テキストと内容概要を出力してください。テキストを読み取れない PDF は内容を読み取って文字起こししてください。」
- **S4（妥当性評価）**: 「入力データに基づき OA の各拒絶理由が適切か厳密に検討する。
  - 新規性・進歩性: 本願の全クレームを構成要件（文節）ごとに分解し、引用文献の**実際の開示内容**と照合。各構成×各引用文献を 5 段階（5=妥当, 1=全く妥当でない）でチャート評価する。
  - **審査官の主張の弱点・誤りを能動的に特定する**こと。具体的には: ①引用文献が当該構成を**本当に開示しているか**（審査官の過大解釈・拡大解釈の有無）、②クレーム解釈の誤り、③開示されていない構成（欠落要素）の有無、④（進歩性で文献を組合せる場合）**動機付け・組合せの合理性・阻害要因（teaching away）・後知恵（hindsight）**の有無、⑤主引例／副引例の position の妥当性。
  - 各拒絶理由について「審査官が強い点／弱い点」を明示し、反論の足がかりになりうる箇所を列挙する。
  - 記載要件・特許適格性・MPF 等はその内容の妥当性を評価。
  - 全評価をチャート＋根拠とともに出力。対象国（JP/US/EP/WO/CN）の審査基準を考慮する。」
- **S6（応答方針）★本アプリの最重要機能。特に新規性・進歩性を詳細に検討**:
  「各拒絶理由を全て解消して**特許査定（許可）を得られる範囲内で、可能な限り広いクレーム**となる応答方針を検討する。これが本検討の最優先目的である。
  - **大前提**: 不必要にクレームを狭めない。**拒絶理由を覆せる（許可が得られる）範囲内で最も広い権利範囲**を狙う。そのために、まず意見書での反論で覆せないかを検討し、補正が必要な場合も限定は必要最小限にとどめる。
  - **狙いどころの特定**: S4 の妥当性評価結果を踏まえ、**引用文献の実際の記載内容と OA の主張を精査**し、審査官の主張のミス・弱点（過大解釈、欠落要素、クレーム解釈の誤り、動機付けの欠如、阻害要因、後知恵 等）を特定して、そこを突く反論・補正を組み立てる。
  - **方針の組み立て方**:
    - 審査官の判断が**明らかに妥当でない**拒絶理由 → 補正は最小限（or 無補正）にして**意見書での反論を主軸**にし、広い権利範囲を維持する。
    - 審査官の判断が**妥当**な拒絶理由 → それを解消する**必要最小限の補正**を行う。追加する限定はクレームを最も狭めないものを選ぶ。
  - **出力要件**:
    - 新規性・進歩性の拒絶理由については、**少なくとも3案**を提示。各案は「広い（攻める）〜狭い（手堅い）」の幅を持たせ、それぞれについて (1)どの拒絶理由にどう対応するか、(2)突く審査官の弱点・法的/事実的根拠、(3)補正の有無と概要（追加する限定）、(4)結果として得られるおおよその権利範囲（広狭）、(5)許可獲得の確度とリスク、を明示する。
    - 記載要件等の拒絶理由については**確実に解消できる案を1案**用意する。
    - **全く補正しない方針案は作らない／過度に限定しすぎる方針案も提案しない**。
    - 審査官が許可可能な請求項を提示している場合は、それに従う案も1つ用意する。
    - 対象国（JP/US/EP/WO/CN）の補正運用・主張の作法を考慮する。
  - 各案の**長所・短所・推奨度**を比較できる形で提示し、ユーザーが選択しやすくする。」
- **S8（代表補正）★重要。必要最小限の補正を徹底**:
  「決定した応答方針に基づき、代表請求項の補正案を検討する。**良い方針でも余計な補正があれば OA 応答は台無しになる**ため、補正は**拒絶理由を解消するために必要最小限の追記・修正**に厳格にとどめる。
  - 追加する文言・限定は、拒絶理由を覆すのに**本当に必要なものだけ**に限る。不要な構成・余計な修飾・過剰な限定を**追加しない**。
  - 追加する一語一語がクレームを狭め、包袋禁反言（エストッペル）のリスクを生むことを意識し、語を増やさない方向で設計する。
  - 同じ拒絶理由を解消できる補正が複数あり得る場合は、**最も権利範囲を狭めない補正**を優先する。
  - 現クレームがあればそれを基準にする。全拒絶理由に対応。新規事項追加（new matter）を避け、補正の根拠（明細書の対応箇所）を示す。
  - 対象国（JP/US/EP/WO/CN）の補正ルールに則る。
  - まず代表的に補正する請求項のみを提示し、補正箇所（削除・追記）をハイライト。可能なら最小限度の異なる3案を提示し、各案の権利範囲の広狭を比較できるようにする。」
- **S10（全文補正）**: 「決定した代表補正案に基づきクレームの全文補正案を作成。**必要最小限の補正に徹し**（S8 の原則を全クレームに適用）、不要な限定を追加しない。現クレームがあれば基準に。全拒絶理由に対応。新規事項追加を避ける。対象国（JP/US/EP/WO/CN）の補正ルールに則る。補正箇所（削除・追記）をハイライト。」
- **S12（意見書）**: 「全文補正案に基づき意見書案を作成。全拒絶理由に対応。対象国のルールに則り主張。**できる限り明細書の記載から抜粋して主張し、エストッペルで制限がかからない主張**にする。」
- **S14（書面出力）**: 「作成済みの全文補正案・意見書案・これまでのデータに基づき、正式な補正書・意見書・見解書の**構造化テキスト**（見出し・本文）を出力。**補正書・意見書は対象国（JP/US/EP/WO/CN）のフォーマット・記載ルールに従う**。見解書には本願概要・OA概要・引用文献概要・妥当性評価結果・複数応答方針案・最終決定方針を含める。」※実ファイル(.docx)生成はアプリ側が行う。

## 12. 非機能要件
- **セキュリティ**: API キー秘匿（§7.1）、RLS 強制、Storage 署名付き URL、認証必須。
- **性能**: ストリーミング表示。長文生成でも UI が固まらない。
- **コスト**: 高トークン消費が前提。**抽出テキストの Supabase 保存・使い回し（PDF を毎回送らない）を主軸**にコスト削減する（§7.5）。実行時は既定 Sonnet（重要ステップのみ Opus に切替可）。プロンプトキャッシュは**オートラン時のみ採用してよい**（連続実行で有効、§7.5/§7.10）。利用状況を把握できると望ましい。
- **可用性 / 信頼性**: API 失敗時のリトライ・明示エラー。途中状態は DB に保存され再開可能。
- **保守性**: システムプロンプト・モデル設定・各国ルールは定数/設定として分離。

## 13. セキュリティ・機密保持
- **個人アカウント利用**。アップロードは**主に公知（公開済み）情報で機密情報を含まない**前提のため、**ZDR は不要**。
- API キー秘匿（§7.1）、RLS 強制、Storage 署名付き URL、認証必須。
- ログに文書本文を残さない。エラーログは最小限に。
- 前提が変わる場合（未公開・機密書類を扱う）はデータ取り扱いを再検討（§7.8）。

## 14. 開発フェーズ・マイルストーン（タスク分割の指針）

> Claude Code はこの順序を基にタスクを分割し、優先度順に実装すること。各タスクは「縦に薄く通す」ことを優先。

1. **環境構築**: Next.js(App Router)+TS+Tailwind 雛形、Vercel デプロイ確認、`.env`（`ANTHROPIC_API_KEY` 等）。
2. **Supabase 接続**: クライアント/サーバーの初期化、Auth（Google + Email/Pass）、`profiles` 自動作成。
3. **DB & Storage & RLS**: §8 のスキーマ、private バケット、RLS ポリシー。
4. **API 疎通**: サーバー側 Route Handler で Claude にストリーミングで 1 往復。
5. **シェル UI**: 2カラム（左サイドバー固定＋右チャット）、固定ヘッダ、固定入力枠。Claude Design のモックを反映。
6. **案件 CRUD**: 新規作成・リスト・再開・削除・**お気に入り登録/解除**（三点メニュー）。
7. **Step1-3（MVP の核）**: アップロード→Storage 保存→**Files API へアップロードして file_id 保存**→解析→要約表示→`extracted_text` 保存。ここで縦の一本が通る。
8. **Step4-5**: 妥当性評価（チャート表示）。
9. **Step6-7 / 8-9 / 10-11 / 12-13**: 方針→代表補正→全文補正→意見書。各ステップで追問可。**S6/S8 は §11 の強化プロンプトを厳守**。
10. **Step14**: 構造化テキスト→`docx` 生成→DL。
11. **オートラン**: Step2→指定ステップの連続実行オーケストレータ（§7.10）。Step群が揃ってから実装。
12. **横断改善**: コンテキスト管理・（オートラン時の）プロンプトキャッシュ・エラー処理・レスポンシブ・UI 磨き込み。

各ステップ完了時に **git commit**。Phase 完了ごとに Vercel で動作確認。

## 15. 決定事項（PO 確定済み）

> 以下は PO により確定済み。Claude Code はこれに従って実装する。

1. **機密性 / ZDR**: 個人アカウント利用、アップロードは主に公知情報で機密なし → **ZDR 不要・クラウド API 利用可**（§7.8, §13）。
2. **利用形態**: **MVP は個人利用**（本人のみ参照、RLS で owner 限定）。将来的に共有を検討。
3. **実行時モデル**: **既定は全ステップ Sonnet**（`claude-sonnet-4-6`）。重要ステップ（S4/S6/S8/S10）は設定で Opus に切替可（§7.2）。
4. **対象国**: **JP / US / EP / WO / CN**。補正・主張は各国ルールをある程度考慮（§4, §11）。
5. **ページ超過**: 100 ページ超文書は**分割して読み込んで可**（§7.3）。
6. **「進む」判定**: **自由入力で判定**（確認ボタンは設けない）。オートラン指示も自由入力で受ける（§10, §7.10）。
7. **Word 体裁**: 社内テンプレ無し。補正書・意見書は**各国フォーマットルールに従う**（§7.7, §11-S14）。
8. **オートラン**: Step1 から指定ステップ（標準は方針案=Step7）まで**ユーザー確認なしで連続実行**できる。各ステップのフル分析は省略しない（§7.10）。
9. **お気に入り**: サイドバー三点メニューから案件のお気に入り登録/解除が可能（§8, §9.2）。

### 残課題（実装しながら詰める）
- 各国（特に EP/CN）補正ルールの精度をどこまで作り込むか（Phase2 以降で段階的に）。
- オートランの停止点 UI（どのステップまで進めるかの指定方法）の具体化。

## 16. 受け入れ基準（MVP / Phase1 の Done 定義）
- Google or Email でログインでき、ログアウトもできる。

## 17. 用語集
- **OA / オフィスアクション**: 特許庁からの拒絶理由通知。
- **本願**: 審査対象の自社出願。**引用文献**: 審査官が拒絶根拠として挙げる先行技術。
- **補正書**: クレーム等を補正する書面。**意見書**: 拒絶理由への反論・釈明書面。**見解書**: 本ツールが出す検討まとめ（社内向け）。
- **エストッペル（包袋禁反言）**: 審査経過での主張が後の権利解釈を制限する原則。意見書では主張しすぎを避ける理由。
- **MPF**: Means-plus-function クレーム（米国）。
- **RLS**: Row Level Security（Supabase/Postgres の行単位アクセス制御）。
- **ZDR**: Zero Data Retention（応答後にデータを保持しない設定/契約）。
- **プロンプトキャッシュ**: 反復利用する大きな入力をキャッシュしコスト/レイテンシを下げる機能。
- **Vertical slice**: 機能を縦に薄く端から端まで通す開発の進め方。
